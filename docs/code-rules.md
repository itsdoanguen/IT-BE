# IT-BE Code Rules and Standards

A comprehensive guide for all developers contributing to the IT-BE backend system.

**Version**: 1.0  
**Status**: Strict - enforcement required for all PRs  
**Last Updated**: 2026-04-13

---

## 1. Architecture and Module Structure

### 1.1 Module-Based Organization

All features must be implemented as **self-contained Django modules** in `modules/`.

```
modules/<feature_name>/
├── models/
│   └── __init__.py              # All model definitions for this module
├── serializers.py               # All serializers
├── views.py                     # All viewsets and custom views
├── urls.py                      # Route registration via DefaultRouter
├── tests.py                     # APITestCase tests
├── admin.py                     # Admin registration (fill even if empty)
├── apps.py                      # Auto-generated by Django
├── migrations/                  # Auto-generated by Django
└── __init__.py                  # Empty
```

**Why this structure:**
- Clear ownership boundary per feature
- Easy to locate code for a domain
- Testability in isolation
- Reusable modules can be shared between projects

**Violation:** Do NOT create subfolders like `views/`, `models/views.py`, or feature branches inside modules.

---

### 1.2 Layer Responsibilities

**models/**: Domain entities and logic
- Define models with all DB fields
- Use model validators only for constraints that apply always
- Use custom managers for complex queries

**serializers.py**: Data transformation and validation
- Define serializers for API requests/responses
- Use `to_representation()` for API response mapping
- Implement field validation in `validate_<field>()` or `validate()`

**views.py**: HTTP endpoint logic
- Define viewsets with get_permissions() for role checks
- Implement get_queryset() for filtering based on request context
- Raise ValidationError for bad request params (→ HTTP 400)
- Keep business logic minimal; delegate to services

**services/**: Domain business logic (if needed)
- Complex calculations and workflows
- External API calls and integrations
- Data aggregations and reporting
- Place in `services/<module_name>_service.py`

**urls.py**: Route registration
- Use DefaultRouter for CRUD endpoints
- Register viewsets with consistent `basename`
- Do NOT inline views here

---

### 1.3 Shared Folders

- **config/**: settings.py, urls.py, wsgi.py, asgi.py (project-wide config)
- **services/**: Cross-module business logic (e.g., `accounts_service.py`, `candidate_matching_service.py`)
- **utils/**: Shared utility functions
- **core/**: Shared base classes, permissions, utilities
- **docs/**: Documentation, plans, contracts, rules (this file belongs here)

## 2. Naming Conventions

### 2.1 Model Names (PascalCase, English)

**Rule**: English class names, domain-meaningful, singular noun.

✅ CORRECT:
```python
class HoSoUngVien(models.Model):           # Candidate Profile
class TinTuyenDung(models.Model):          # Job Posting
class UngTuyen(models.Model):              # Application/Submission
class DanhGia(models.Model):               # Review/Rating
```

❌ WRONG:
```python
class CandidateProfile(models.Model):      # Mix Vietnamese domain term with English class
class ho_so_ung_vien(models.Model):        # Lowercase
class Candidate(models.Model):             # Too generic, loses domain context
```

### 2.2 Model Fields (snake_case, Vietnamese where applicable)

**Rule**: Use Vietnamese field names to map domain terminology. Use db_column for FK IDs if needed.

✅ CORRECT:
```python

    so_dien_thoai = models.CharField(max_length=20, null=True, blank=True)

    cap_nhat_luc = models.DateTimeField(auto_now=True)

    class Meta:
```

### 3.1 All Imports Must Be Absolute

**Rule**: Use absolute imports only. Never use relative imports (`.` or `..`).

✅ CORRECT:
```python

from modules.accounts.models import NguoiDung
```

❌ WRONG:
```python
from . import models                       # Relative

```

### 3.2 Import Order (PEP 8 + Django)

Order:
1. Python standard library (os, sys, collections, etc.)
2. Third-party (django, rest_framework, etc.)
3. Local imports (modules.*, services.*, utils.*, core.*)

Within each group, alphabetical order.

✅ CORRECT:
```python

from django.db.models import Q
from django.utils import timezone

from rest_framework.exceptions import ValidationError

```

❌ WRONG:
```python

from rest_framework import viewsets
from django.db.models import Q                 # Django after local, out of order
import os
```

## 4. Django/DRF Patterns

### 4.1 Role-Based Access Control via get_permissions()

**Rule**: Override `get_permissions()` to enforce role checks; never skip auth in list/retrieve.

✅ CORRECT:
```python
from core.permissions import IsEmployer, IsCandidate

class HoSoUngVienViewSet(viewsets.ModelViewSet):
    def get_permissions(self):
        if self.action in ["list", "retrieve"]:
            return [IsEmployer()]                # Only employers can view candidates
        elif self.action == "update":
            return [IsCandidate()]               # Only own candidate can update
        else:
            return [permissions.IsAuthenticated()]

class TinTuyenDungViewSet(viewsets.ModelViewSet):
    public_actions = {"list", "retrieve"}

    def get_permissions(self):
        if self.action in self.public_actions:
            return [permissions.AllowAny()]      # Public read
        return [permissions.IsAuthenticated()]   # Auth required for modify
```

❌ WRONG:
```python
class HoSoUngVienViewSet(viewsets.ModelViewSet):
    # No permission override — uses default IsAuthenticated everywhere
    # Allows any authenticated user to see all candidates
    pass

class TinTuyenDungViewSet(viewsets.ModelViewSet):
    permission_classes = [IsAuthenticated]      # Static, not role-based
```

### 4.2 Query Parameter Filtering and Validation

**Rule**: Explicit validation in `get_queryset()` or a dedicated method. Raise `ValidationError` for 400 errors.

✅ CORRECT:
```python
class TinTuyenDungViewSet(viewsets.ModelViewSet):
    default_status = "dang_mo"
    allowed_statuses = ["dang_mo", "da_dong"]

    def get_queryset(self):
        queryset = TinTuyenDung.objects.select_related("cong_ty").order_by("-tao_luc")
        query_params = self.request.query_params

        # Status validation
        status = query_params.get("trang_thai", "").strip() or self.default_status
        if status not in self.allowed_statuses:
            raise ValidationError({"trang_thai": "Gia tri khong hop le."})

        # Salary range validation
        salary_min = query_params.get("salary_min", "").strip()
        if salary_min:
            try:
                min_salary = Decimal(salary_min)
                queryset = queryset.filter(luong_theo_gio__gte=min_salary)
            except (InvalidOperation, ValueError) as exc:
                raise ValidationError({"salary_min": "Must be a valid number"}) from exc

        # Keyword search
        keyword = query_params.get("q", "").strip()
        if keyword:
            queryset = queryset.filter(Q(tieu_de__icontains=keyword) | Q(noi_dung__icontains=keyword))

        return queryset
```

❌ WRONG:
```python
def get_queryset(self):
    # Silently ignores bad params
    queryset = TinTuyenDung.objects.all()
    keyword = self.request.query_params.get("q")
    if keyword:
        queryset = queryset.filter(tieu_de__icontains=keyword)
    # No status validation, no salary validation
    return queryset
```

### 4.3 Response Shaping via to_representation()

**Rule**: Use `to_representation()` in serializers to map Vietnamese fields to English API fields. Keep both raw and transformed data if useful.

✅ CORRECT:
```python
class TinTuyenDungSerializer(serializers.ModelSerializer):
    class Meta:
        model = TinTuyenDung
        fields = "__all__"

    def to_representation(self, instance):
        data = super().to_representation(instance)
        raw_data = dict(data)

        # Map Vietnamese to English for API contract
        data["title"] = data.get("tieu_de") or ""
        data["description"] = data.get("noi_dung") or ""
        data["salary"] = self._format_salary(data.get("luong_theo_gio"))
        data["status"] = self._format_status(data.get("trang_thai"))
        data["location"] = data.get("dia_diem_lam_viec") or ""
        data["raw"] = raw_data                   # Keep original if needed

        return data

    @staticmethod
    def _format_salary(salary_value):
        if salary_value in (None, ""):
            return "Not updated"
        return f"{salary_value} / hour"

    @staticmethod
    def _format_status(status_value):
        status_map = {
            TinTuyenDung.TrangThai.DANG_MO: "Open",
            TinTuyenDung.TrangThai.DA_DONG: "Closed",
        }
        return status_map.get(status_value, "Unknown")
```

❌ WRONG:
```python
class TinTuyenDungSerializer(serializers.ModelSerializer):
    class Meta:
        model = TinTuyenDung
        fields = ["tieu_de", "noi_dung", "luong_theo_gio"]  # Raw Vietnamese fields only
    # No transformation; API returns Vietnamese field names directly
```

### 4.4 QuerySet Optimization with select_related() and prefetch_related()

**Rule**: Use `select_related()` for ForeignKey/OneToOneField; `prefetch_related()` for reverse relationships.

✅ CORRECT:
```python
class TinTuyenDungViewSet(viewsets.ModelViewSet):
    def get_queryset(self):
        return TinTuyenDung.objects.select_related("cong_ty").prefetch_related("ung_tuyen").order_by("-tao_luc")

class UngTuyenViewSet(viewsets.ModelViewSet):
    def get_queryset(self):
        return UngTuyen.objects.select_related("tin", "ung_vien").order_by("-tao_luc")
```

❌ WRONG:
```python
def get_queryset(self):
    return TinTuyenDung.objects.all()       # N+1 queries when accessing cong_ty in serializer
```

### 4.5 Use Q Objects for Complex Filters

**Rule**: Use Django's Q object for OR/AND logic; avoid duplicating filter logic.

✅ CORRECT:
```python
from django.db.models import Q

queryset = queryset.filter(
    Q(tieu_de__icontains=keyword) | Q(noi_dung__icontains=keyword) | Q(ky_nang__icontains=keyword)
)
```

❌ WRONG:
```python
queryset = queryset.filter(tieu_de__icontains=keyword)
queryset = queryset.filter(noi_dung__icontains=keyword)  # AND condition, not OR
```

### 4.6 Custom Serializer Methods

**Rule**: Implement custom create/update/validate methods in serializer, not view.

✅ CORRECT:
```python
class NguoiDungSerializer(serializers.ModelSerializer):
    password = serializers.CharField(write_only=True, required=True)

    class Meta:
        model = NguoiDung
        fields = ["id", "email", "password", "vai_tro"]
        read_only_fields = ["id"]

    def create(self, validated_data):
        password = validated_data.pop("password")
        return NguoiDung.objects.create_user(password=password, **validated_data)

    def validate_email(self, value):
        if NguoiDung.objects.filter(email=value).exists():
            raise serializers.ValidationError("Email already registered")
        return value
```

❌ WRONG:
```python
class NguoiDungSerializer(serializers.ModelSerializer):
    # No custom create; relies on view to call create_user()
    pass

# In view:
def perform_create(self, serializer):
    password = serializer.validated_data.pop("password")
    NguoiDung.objects.create_user(...)
```

## 5. Error Handling

### 5.1 HTTP Status Codes

**Rule**: Use correct status codes for different scenarios.

| Scenario | Status Code | Example |
|----------|-------------|---------|
| Resource created | 201 Created | POST /api/v1/candidates |
| Resource updated | 204 No Content | PATCH /api/v1/candidates/{id} |
| Bad query params | 400 Bad Request | Invalid sort parameter |
| Unauthenticated | 401 Unauthorized | Missing JWT token |
| Wrong role | 403 Forbidden | Non-employer accessing candidate list |
| Resource not found | 404 Not Found | GET /api/v1/candidates/999 |
| Conflict/duplicate | 409 Conflict | Email already exists |
| Empty search result | 200 OK | Return empty array, not error |

### 5.2 Error Response Format

**Rule**: Always return structured error JSON with field-level details.

✅ CORRECT:
```python
# In views.py:
if status not in allowed_statuses:
    raise ValidationError({"trang_thai": "Invalid status value"})
    # Returns: {"trang_thai": "Invalid status value"} with HTTP 400

# For auth failures:
# DRF automatically returns {"detail": "Authentication credentials..."} with HTTP 401
```

**Error response for validation failure (400):**
```json
{
  "salary_min": "Must be a valid number",
  "availability_slots": "Invalid JSON format"
}
```

### 5.3 Partial vs Full Validation

**Rule**: PATCH accepts partial updates; PUT requires full object. Use serializer's `partial=True`.

✅ CORRECT:
```python
# In generic viewset (auto-handled):
# PATCH /api/v1/candidates/{id} → partial=True (can update one field)
# PUT /api/v1/candidates/{id} → partial=False (must provide all fields)
```

## 6. Testing

### 6.1 Test File Location and Organization

**Rule**: Each module has a `tests.py` file using Django's APITestCase.

✅ CORRECT:
```
modules/profiles/
├── tests.py
```

```python
# modules/profiles/tests.py
from rest_framework.test import APITestCase
from rest_framework import status
from modules.profiles.models import HoSoUngVien
from modules.accounts.models import NguoiDung

class HoSoUngVienViewSetTests(APITestCase):
    def setUp(self):
        self.user = NguoiDung.objects.create_user(
            email="candidate@example.com",
            password="password123",
            vai_tro=NguoiDung.VaiTro.UNG_VIEN,
        )
        self.profile = HoSoUngVien.objects.create(
            ung_vien=self.user,
            ho_ten="Nguyen Van A",
            ky_nang="Python, Django",
        )

    def test_list_candidates_requires_employer_role(self):
        self.client.login(email="candidate@example.com", password="password123")
        response = self.client.get("/api/v1/candidates")
        self.assertEqual(response.status_code, status.HTTP_403_FORBIDDEN)

    def test_list_candidates_returns_data_with_pagination(self):
        employer = NguoiDung.objects.create_user(
            email="employer@example.com",
            password="password123",
            vai_tro=NguoiDung.VaiTro.CONG_TY,
        )
        self.client.force_authenticate(user=employer)
        response = self.client.get("/api/v1/candidates?page=1&limit=10")
        self.assertEqual(response.status_code, status.HTTP_200_OK)
        self.assertIn("results", response.data)
```

❌ WRONG:
```
modules/profiles/
└── tests/                    # Separate directory
    ├── test_models.py
    ├── test_views.py
    └── test_serializers.py
```

✅ CORRECT:

### 6.3 Test Coverage Checklist

Each API endpoint test file must cover:
- ✅ Authentication (401 for unauthenticated)
- ✅ Authorization (403 for wrong role)
- ✅ Happy path (200 response with correct data)
- ✅ Invalid input (400 for malformed params)
- ✅ Not found (404 for nonexistent resource)
- ✅ Empty result (200 with empty array)
- ✅ Filtering (independent and combined filters)
- ✅ Sorting (all sort options)
- ✅ Pagination (boundary cases)

## 8. Security Guidelines

### 8.1 Authentication and Authorization

**Rule**: Never skip authentication on data-sensitive endpoints.

✅ CORRECT:
```python
def get_permissions(self):
    if self.action in ["list", "retrieve"]:
        return [IsEmployer()]   # Role-based guard
    return [IsAuthenticated()]
```

❌ WRONG:
```python
permission_classes = [AllowAny]  # No auth on candidate list
```

### 8.2 Sensitive Data

**Rule**: Never return raw passwords, tokens, or sensitive user data in API responses.

✅ CORRECT:
```python
class NguoiDungSerializer(serializers.ModelSerializer):
    password = serializers.CharField(write_only=True)

    class Meta:
        model = NguoiDung
        fields = ["id", "email", "vai_tro"]  # No password field in response
```

### 8.3 Input Validation

**Rule**: Validate all user input explicitly. Use model and serializer validators.

✅ CORRECT:
```python
class DanhGia(models.Model):
    diem_so = models.PositiveSmallIntegerField(
        validators=[MinValueValidator(1), MaxValueValidator(5)]
    )

class DanhGiaSerializer(serializers.ModelSerializer):
    def validate_diem_so(self, value):
        if value < 1 or value > 5:
            raise serializers.ValidationError("Rating must be between 1 and 5")
        return value
```

## 14. Frequently Asked Questions

**Q: Can I use relative imports (`from .models import`)?**  
A: No, always use absolute imports. Easier to refactor and understand.

**Q: Should I add all models to Django admin?**  
A: Yes, keep `admin.py` filled even if registration is minimal. Makes future admin access easier.

**Q: What if my feature needs complex business logic?**  
A: Put it in `services/<module>_service.py` as standalone functions or classes. Call from views.

**Q: Do I need type hints?**  
A: In services and complex functions: yes. In models/serializers/views: no, not required but appreciated.

**Q: How do I handle transactions?**  
A: Use Django's `@transaction.atomic` decorator for multi-step operations that need rollback.

**Q: What's the max response payload size?**  
A: Aim for < 1MB per request. Use pagination for large result sets.
